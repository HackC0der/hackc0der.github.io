2025RCTF的一道Python PWN题。前段时间学习了两道Python PWN题（CPython PWN | HackC0der’s Blog）主要是针对Python一切皆对象的特性进行利用。

在Python中，字符串（String）是一种不可变的数据类型。这意味着一旦创建了一个字符串对象，它的值就不能被改变。如果尝试修改字符串的内容，Python会创建一个新的字符串对象，而不是修改原有的字符串。本题基于字符串对象，利用ctype等底层接口实现了一个可边长的字符串，肯定会出现一些问题。

Python的字符串实现

  • type PyASCIIObject

  • type PyCompactUnicodeObject

  • type PyUnicodeObject

    These subtypes of PyObject represent a Python Unicode object. In almost all cases, they shouldn’t be used directly, since all API functions that deal with Unicode objects take and return PyObject pointers.

Python-3.12.4/Include/cpython/unicodeobject.h

结构体 核心特征 适用场景
PyASCIIObject 最基础的结构。用于存储纯 ASCII 字符（码点 U+0000 到 U+007F）。数据紧随结构体之后，采用单内存块的紧凑分配方式。state.ascii和 state.compact标志位均设为 1。 例如 "hello"这样的纯英文文本。
PyCompactUnicodeObject 继承自 PyASCIIObject。用于存储包含非 ASCII 字符的字符串。增加了 utf8_length和 *utf8指针，用于缓存字符串的 UTF-8 编码形式，提升编码效率。同样采用紧凑分配。 例如包含中文 "你好"或拉丁字符 "café"的字符串。
PyUnicodeObject 继承自 PyCompactUnicodeObject。主要被 Unicode 子类 或称为 传统字符串（Legacy String） 的对象使用。它包含一个 data联合体（union），明确指向不同编码（如 UCS1、UCS2、UCS4）的数据缓冲区。通常使用两个独立的内存块（结构体一块，数据一块）。 通常由用户自定义的字符串子类创建。

关键机制：state位域与字符存储宽度：

PyASCIIObject中的 state位域是控制字符串存储和行为的关键，它使用位字段来节省内存。

  • interned:2: 指示字符串是否被 驻留（interned）。驻留是一种优化机制，对内容相同的字符串只保留一个副本，通过 is关键字可以验证其身份一致性。其值可以是 0（未驻留）、1（已驻留）、2（已驻留且永生）等。
  • kind:3: 这是最重要的字段之一，它定义了每个字符在内存中占用的字节数，直接影响内存使用和能表示的字符范围。 PyUnicode_1BYTE_KIND (1): 每个字符占 1 字节。可表示 Latin-1 编码范围内的所有字符（U+0000 到 U+00FF）。如果字符串中所有字符都在 ASCII 范围内（U+0000 到 U+007F），则 ascii标志为 1。 PyUnicode_2BYTE_KIND (2): 每个字符占 2 字节。可表示基本多文种平面（BMP）的所有字符（U+0000 到 U+FFFF），包括绝大多数常用汉字。 PyUnicode_4BYTE_KIND (4): 每个字符占 4 字节。可以表示整个 Unicode 范围内的字符（U+0000 到 U+10FFFF），包括一些表情符号（emoji）和罕见字符。
  • compact:1: 表明字符串对象是否采用紧凑内存布局（结构体和字符数据在同一块连续内存中）。PyASCIIObject和 PyCompactUnicodeObject通常是紧凑的（值为 1），而 PyUnicodeObject通常不是（值为 0）。
  • ascii:1: 表明字符串是否仅包含 ASCII 字符。这是基于字符码点的快速判断标志。

二、永生对象

回到题目，我们输出一些变量数值以便调试：

简单来说，似乎"a"被“驻留”了（？），所以每次 new a时，data都指向同一个字符串对象（data id相同）；而 "123_abc"没有被“驻留”，每次 new 123_abc都会创建一个新的字符串对象。下面是ai的解释：

Python 的字符串驻留机制主要为了优化性能和内存，但并非所有字符串都会被驻留。它主要发生在以下情况：

  1. 编译期确定的字符串：在代码编译阶段就能确定的字符串字面量更可能被驻留。你的代码中，通过 input().split()动态生成的字符串属于运行时创建，驻留条件更为严格。
  2. 符合标识符规则的字符串：由字母、数字和下划线 _组成的字符串，因为它们很可能被用作变量名、函数名等标识符，重复使用率高。你的 "123_abc"虽然符合此规则，但因为是运行时动态输入，可能未被驻留。
  3. 长度很短的字符串：长度為 0 或 1 的字符串几乎总是被驻留。这就是为什么单字符 "a"被驻留了

然而，通过gdb调试我们可以发现奇怪的一幕——“a”的state.interned == 0——但是state.statically_allocated = 1

——这是为什么，难道不是驻留机制吗？为什么没有置位？看看AI的回答与总结：

永生对象（Immortal Objects）是 CPython 解释器内部一种特殊的对象，其核心特性是引用计数永远不会减为零，因此也永远不会被垃圾回收器销毁。这项特性主要是为了支持多解释器环境而引入的，旨在解决特定场景下的性能和安全问题。

在 CPython 中，每个对象都有一个引用计数字段（通常是 32 位或 64 位无符号整数）。永生对象的实现方式是将这个引用计数值设置为一个特定的、不可能通过正常引用操作达到的极大值（例如 32 位无符号整数的最大值 0xFFFFFFFF）。

这样一来，Py_INCREF()和 Py_DECREF()这类管理引用计数的宏在操作对象时，会先检查其引用计数值。如果发现该值已经是这个特殊的极大值，就会跳过实际的增加或减少操作，因为解释器知道这个对象是永生的，无需管理其生命周期

目前，Python 内置的一些不可变对象通常是永生的，例如：

  • None, True, False
  • 小整数（通常是一个预先定义好的范围，如 -5 到 256）
  • 一些常用的字符串（空字符串、单个字符等也可能被驻留并设置为永生）

永生对象不通过常规的“驻留字典”来管理，因此它的 interned标志位是 0，但它通过更底层的“静态分配”机制实现了全局唯一，从而被所有变量共享。

下面这个表格清晰地展示了普通驻留对象与你遇到的永生对象在内部机制上的核心区别：

特性 普通驻留对象 永生对象
state.interned标志位 SSTATE_INTERNED_MORTAL(通常为 1) SSTATE_NOT_INTERNED(0)
state.statically_allocated标志位 0 1
管理机制 由全局 interned字典管理 在 Python 解释器初始化时被静态分配，并永久存在
生命周期 可被垃圾回收（在清理驻留池时） 永生，与 Python 解释器生命周期相同，永不释放
典型例子 运行时创建的符合规则的标识符 短小的拉丁字母字符、数字、None、True、False等

特性参考：PEP 683 – 不朽对象，使用固定引用计数 | peps.python.org - Python 增强提案

Python的字符串是静态不可修改的，本题则提供了一个能够从底层修改字符串的操作，因此不由得进行思考：

永生对象意味着同一个变量在许多位置被引用，而如果永生对象发生了变化，则会导致全局的该永生对象的引用位置的值均发生改变。
