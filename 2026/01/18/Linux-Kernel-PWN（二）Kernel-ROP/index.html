ROP 即返回导向编程（Return-oriented programming）——通过复用代码片段的方式控制程序执行流。

内核态的 ROP 与用户态的 ROP 一般无二，只不过利用的 gadget 变成了内核中的 gadget，所需要构造执行的 ropchain 由 system("/bin/sh") 变为了 commit_creds(&init_cred) 或 commit_creds(prepare_kernel_cred(NULL))，当我们成功地在内核中执行这样的代码后，当前线程的 cred 结构体便变为 init 进程的 cred 的拷贝，我们也就获得了 root 权限，此时在用户态起一个 shell 便能获得 root shell。

一、状态保存

通常情况下，我们的 exploit 需要进入到内核当中完成提权，而我们最终仍然需要着陆回用户态以获得一个 root 权限的 shell，因此在我们的 exploit 进入内核态之前我们需要手动模拟用户态进入内核态的准备工作——保存各寄存器的值到内核栈上，以便于后续着陆回用户态。

init_module

init_module() 函数是 Linux 内核模块的初始化函数之一。在编写内核模块时，init_module() 用于执行在模块加载时需要初始化的任务，例如注册设备、初始化资源、配置网络接口等操作。

proc_create 是 Linux 内核中的一个函数，用于创建一个新的 /proc 文件系统条目。这个函数常用于内核模块中，以便在 /proc 文件系统下创建一个新的文件，使得用户空间程序可以通过这个文件与内核模块进行交互。

  • name: 这是要创建的 /proc 文件的名称。它是一个字符串，表示文件的名称。
  • mode: 这是文件的权限模式，通常使用 S_IRUGO、S_IWUSR 等宏来设置读、写权限等。umode_t 是一个表示文件模式的类型。
  • parent: 这是一个指向父目录条目的指针。如果为 NULL，则在根目录下创建文件。
  • fops: 这是一个指向 file_operations 结构的指针，包含了对这个文件的操作函数的定义，例如打开、读取、写入等操作。

5.ROP-Chain

vmlinux 则是静态编译，未经过压缩的 kernel 文件。我们可以从 vmlinux 中找到一些 gadget，我们先把 gadget 保存下来备用。

注意：找gadget的偏移，不能从带符号的vmlinux里面找gadget,要从bzImage里面提取之后再找gadget，先用extract-vmlinux*提取文件出来，再用**ropper*找gadegt

带符号的vmlinux包含符号表、调试段、未优化的代码段，这些内容不会被加载到系统内存中运行；而系统实际运行的是bzImage解压后的内核二进制，经过了编译优化、段裁剪、地址重定位，和vmlinux的代码布局、字节内容、地址偏移完全不同。

如果直接在vmlinux里找 Gadget，得到的偏移地址是基于调试文件的，和实际内存中内核的 Gadget 地址完全不匹配，利用时会直接跳错位置，攻击失败。

补充说明
  1. 关键内核函数

  • prepare_kernel_cred(pid)：生成一个新的 cred 权限结构，参数为0时表示生成root 用户（uid=0/gid=0） 的 cred 结构，函数返回值（存在rax中）是 root cred 的地址；

  • commit_creds(cred)：将当前进程的权限结构替换为参数传入的 cred 结构，调用后当前进程立即获得对应权限（传入 root cred 则直接提权为 root）。

  1. iretq 指令（x86_64 特权级切换核心）

iretq是 x86_64 的64 位中断返回指令，也是内核态切回用户态的标准指令（唯一安全的跨特权级返回方式），执行时会从栈上按固定顺序弹出 6 个值，分别加载到对应的寄存器，栈上的顺序要求严格固定（少一个 / 多一个都会导致内核崩溃）：

  • cs：加载到代码段寄存器（必须是用户态的 cs，即保存的user_cs，最低 2 位 = 3）；
  • ss：加载到栈段寄存器（用户态的 ss，user_ss，最低 2 位 = 3）；
  • rsp：加载到栈指针寄存器（用户态的栈顶，user_sp）；
  • rflags：加载到标志寄存器（用户态的标志位，user_rflags，需保证 IF=1 开启中断）；
  • rip：加载到指令指针寄存器（切回用户态后要执行的代码地址，的 payload 中是binsh，即/bin/sh的执行入口）；
  1. 整体流程

  • 保存用户态：保存用户态寄存器（user_cs/user_rflags/user_sp/user_ss）

  • 漏洞利用：通过内核栈溢出劫持执行流

  • 内核提权：调用内核核心函数prepare_kernel_cred(0)生成 root 权限的 cred 结构，再调用commit_creds()将当前进程的 cred 替换为 root 的（Linux 中cred是进程的权限核心结构，替换后进程直接获得 root）；

  • 恢复寄存器上下文：通过swapgs切换内核 / 用户态的 GS 段（x86_64 中 GS 段用于区分内核 / 用户态的线程数据），为切回用户态做准备；

  • 内核态→用户态切换：通过 x86_64 的中断返回指令 iretq，加载之前保存的用户态寄存器（user_cs/user_rflags/user_sp/user_ss），从内核态切回用户态，执行/bin/sh获得 root shell。

  1. 内核符号表地址映射

kallsyms（内核符号表）是Linux 内核的核心调试 / 符号映射机制，本质是内核维护的一份 「地址 ↔ 内核符号」的映射表 ，其中的 “符号” 包括内核函数名、全局变量名、结构体名等，/tmp/kallsyms（或原生的/proc/kallsyms）就是这份映射表的文本化输出文件，也是内核提权中获取prepare_kernel_cred/commit_creds等函数地址的核心依据。prepare_kernel_cred和commit_creds的符号类型都是大写 T 的导出函数，所以能直接从 kallsyms 中获取地址；

让ai写一个函数，用于读取/tmp/kallsyms，依据传入的sym_name，返回对应符号的地址

  1. swapgs; popfq; ret;

该 Gadget 是x86_64 Linux 内核提权中从内核态切回用户态的必备指令组合，由 3 条连续的 x86_64 汇编指令构成，核心作用是完成内核态到用户态的硬件上下文准备，为后续iretq（中断返回指令）执行跨特权级切换铺路。

指令 核心作用 内核提权中的关键意义
swapgs 交换内核态与用户态的 GS 段寄存器值 x86_64 架构中，GS 段是内核 / 用户态的线程数据段标识：内核态 GS 指向内核线程数据（task_struct指针，进程核心控制结构），用户态 GS 指向用户态线程局部存储（TLS）。切回用户态前必须执行该指令，否则后续iretq会访问错误的内核数据，直接导致内核崩溃（Oops）。
popfq 从栈中弹出 8 字节数据，加载到 RFLAGS 标志寄存器 恢复标志寄存器的合法状态：栈上提前预置了用户态的rflags值（即你之前保存的user_rflags），执行后 RFLAGS 会被重置为用户态的标志位（需保证IF 位（第 9 位）=1，开启外部中断，否则切回用户态后进程无响应）。补充：popfq是 x86_64 对popf的 64 位扩展，专门操作 64 位 RFLAGS 寄存器。
ret 从栈中弹出 8 字节地址，跳转到该地址执行 ROP 链的拼接核心：完成前两条指令后，通过ret指令从栈上取下一个 Gadget / 指令的地址（你的场景中就是iretq; ret），让 ROP 链继续按预设逻辑执行，实现指令流的无缝拼接。
