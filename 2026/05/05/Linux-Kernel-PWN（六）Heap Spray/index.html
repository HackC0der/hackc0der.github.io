[Linux-Kernel-PWN] Heap Spray

堆喷射（heap spraying）指的是一种辅助攻击手法：「通过大量分配相同的结构体来达成某种特定的内存布局，从而帮助攻击者完成后续的利用过程」，常见于如下场景：

  • 你有一个 UAF，但是你无法通过少量内存分配拿到该结构体（例如该 object 不属于当前 freelist 且释放后会回到 node 上，或是像 add_key() 那样会被一直卡在第一个临时结构体上），这时你可以通过堆喷射来确保拿到该 object。
  • 你有一个堆溢出读 / 写，但是堆布局对你而言是不可知的（比如说开启了 SLAB_FREELIST_RANDOM（默认开启）），你可以预先喷射大量特定结构体，从而保证对其中某个结构体的溢出。
  • ……

作为一种辅助的攻击手法，堆喷射可以被应用在多种场景下。

内核密钥相关结构体

为了绕过 kaslr 我们需要 leak 内核中指令的地址从而获得 base addr

这一步是学习 [a3](【PWN.0x02】Linux Kernel Pwn II：常用结构体集合 - arttnba3’s blog) 师傅的相关博客，跟着 5.19.0 linux kernel source code 完整走了一遍，对于利用内核密钥的相关结构体过程有了初步的了解，形成了 自己的博客

__keyctl_read_key() 的主要逻辑是调用 key->type->read(key, buffer, buflen);，在key->type->read 内部实际会比较 buflen 和 upayload->datalen 选择小的数据作为复制（读取）长度。由于buflen 是用户可控，所以如果我们能够改大upayload->datalen这一数据，就可以造成数据泄露。

泄露什么呢？—— key->type->revoke() 时会在内存写上 user_free_payload_rcu

为此我们可以通过释放 key 在内存上留下代码段指针数据，然后通过改大特定 key的 payload->datalen 从而 OOB Read 相邻已释放的 key 内存上的代码段指针。

利用 Heap Spray 分配内存到特定结构体

构建内核密钥时需要传入 description 和 payload 两个参数（这里只考虑 type 为 "user" 的情况）。为了在内核态安全的保存这来自用户态的数据，经历了多次内核内存申请-释放的过程：

  • 按照最大的空间申请临时内存存放 description：description = strndup_user(_description, KEY_MAX_DESC_SIZE);，方式是kmalloc(size, GFP_KERNEL) ，申请大小为 4096
  • 为 payload 申请临时空间：payload = kvmalloc(plen, GFP_KERNEL);，申请大小为传入数据
  • 进入key_create_or_update 函数：key_ref = key_create_or_update(keyring_ref, type, description, payload, plen, KEY_PERM_UNDEF, KEY_ALLOC_IN_QUOTA)
    • 为 description 申请内存长期保存数据：key_alloc 函数中 key->index_key.description = kmemdup(desc, desclen + 1, GFP_KERNEL);，申请大小为 description 长度 +1
    • 为 payload 申请内存长期保存数据：ret = index_key.type->preparse(&prep); ，upayload = kmalloc(sizeof(*upayload) + datalen, GFP_KERNEL);，申请大小为 “结构体user_key_payload的大小 + 传入数据的大小”
  • 释放临时存储 payload 的内存：kvfree_sensitive(payload, plen);
  • 释放临时存储 description 的内存：kfree(description);

值得注意的是：临时 payload 与 长期存储 payload 的内存申请，size 十分靠近。当这个 size 实际申请的是同一个 2^n 大小的堆块时，似乎始终有一个堆块作为 ”摆渡人” ，一直充当临时变量。对于本题目，在构造一个 UAF 之后，释放的 UAF Object 似乎始终充当这个临时的 payload 结构体，无法分配到长期存储的 key->payload 中。

但是这里有一个问题：add_key() 会先分配一个临时的 obj1 拷贝 payload 后再分配一个 obj2 作为 user_key_payload，若我们先分配一个 obj 并释放后再调用 add_key() 则该 obj 不会直接成为 user_key_payload ，而是会在后续的数次分配中都作为拷贝 payload 的临时 obj 存在。

但我们可以通过堆喷将 UAF obj 分配到 user_key_payload，考虑如下流程：

  • 利用题目功能构建 UAF object。
  • 堆喷射 user_key_payload ，UAF obj 作为拷贝 payload 的临时 obj 存在。
  • kmem_cache_cpu 的 slub page 耗光，向 node 请求新的 slub page 分配 user_key_payload ，完成后 UAF obj 被释放并回到 kmem_cache_node。
  • 继续堆喷 user_key_payload ，kmem_cache_cpu 的 slub page 耗光，向 node 请求新的 slub page 分配 user_key_payload 。
  • UAF obj 所在页面被取回，UAF obj 被分配为 user_key_payload 。
  • 利用题目功能再次释放 UAF obj，利用题目功能进行堆喷获取到该 obj，从而覆写 user_key_payload 。

如上流程，通过 Heap Spray 将 UAF Object 分配到 user_key_payload ：

  1. 首先取 UAF obj 作为 临时obj1 ，此时 kmem_cache_cpu 还有 20 个 obj
  2. 持续 add_key，最后 kmem_cache_cpu 只剩一个 obj，也即 UAF obj
  3. 继续一次 add_key：
    • 取 kmem_cache_cpu 最后的 UAF obj 作 临时 obj；
    • 继续取 obj 时将 kmem_cache_cpu 的 slub 换出到 kmem_cache_node 中的 full 链表；
    • 重新申请一个 slub 放到 kmem_cache_cpu 中并取出第一个 obj 作 长期存储 obj ，kmem_cache_cpu 中还剩下 20 个 obj
    • 释放 临时 obj（UAF obj），由于属于 kmem_cache_node 中 full 链表的 slub，所以会让该 slub 从 full 链表中取出放到 partial 链表，同时 UAF obj 挂到该 slub 中。
  4. 持续 add_key，最后 kmem_cache_cpu 只剩下一个 obj
  5. 继续一次 add_key：
    • 取 kmem_cache_cpu 最后的 obj 作 临时 obj；
    • 继续取 obj 时将 kmem_cache_cpu 的 slub 换出到 kmem_cache_node 中的 full 链表；
    • 取 kmem_cache_node 中 partial 链表上的 slub 到 kmem_cache_cpu，该 slub 只有一个 UAF obj
    • 将 UAF obj 取出作为 user_key_payload 存储

实际上，要想将 UAF obj 释放到 user_key_payload ，还可以通过释放 key 从而将原本已有的一个 user_key_payload 占据的内存头插到 kmem_cache_cpu，从而代替 UAF obj 成为 临时 obj。

因此这里至少有三种方式，都能够将 UAF obj 分配到 user_key_payload：

利用题目自带的两次 UAF 调整 free_list

利用 key_revoke 调整 free_list

我们只需要有一个其他的 obj 查到 free_list 头部即可，那为何不申请 key 后释放 payload（user_key_payload）也是可以做到这一点的。要注意的一点是，revoke 过程中，要等 RCU 宽限期结束 才会调用 rcu_free_key_payload 释放 payload，所以我们需要适当延时。

利用 Heap Spary 调整 free_list

这一个方法就是一直在提及的，也是本题主要学习的技巧方式：通过堆喷，利用 slub 在 kmem_cache_cpu 和 kmem_cache_node 之间的换入换出来调整分配结果。

[!Important]

在 Linux user 类型 key 机制中，同一进程 keyring 下，多个 key 使用完全相同的 description（描述名）时，内核会进行查重、关联、状态强校验，会直接影响 key_read 行为甚至导致 key 被判定为非法。
原理细节

  1. add_key 创建 user 类型 key 时，内核会在当前进程 keyring 中按 description 做匹配检索；
  2. 若多个 key description 完全一致，内核会做链表关联与唯一性校验、引用计数关联、有效性联动检查；
  3. 当其中某个 key 被 UAF 篡改 user_key_payload 结构体（尤其是篡改 datalen、破坏 rcu_head）后，内核因同名 key 关联校验机制，会判定该 key 结构非法、状态异常；
  4. 此时调用 key_read 读取该被篡改的同名 key，内核会直接返回 -1（系统调用错误），不会走正常 OOB 读取分支，且极易在错误处理路径触发 key 释放、RCU 回调执行，造成内核空指针崩溃。

Pwn 利用实战影响

  1. heap spray 批量创建 key 时，不能全部写死同一个描述名（如固定 “xxxx”）；
  2. 必须给每一个 key 分配独立唯一的 description，例如 key0、key1、arttnba0 等递增命名；
  3. 唯一描述名可规避内核同名 key 查重与联动校验，让被 UAF 篡改的 key 不会被内核提前标记为非法；
  4. 保证 key_read 能正常读取、返回超出正常 object 大小的值，进入 OOB 泄露分支，不会返回 -1、也不会提前触发 key 释放与 RCU 崩溃。
