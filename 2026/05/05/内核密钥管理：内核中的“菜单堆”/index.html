【PWN.0x02】Linux Kernel Pwn II：0x0A-内核密钥管理：内核中的“菜单堆 - arttnba3’s blog

自 Linux 2.6 起内核引入了 密钥保留服务（key retention service），用以在内核空间存储密钥以供其他服务使用，并提供了用以在用户空间操作密钥的三个新的系统调用，我们主要关注可供利用的部分。

本篇博客参考的内核源码版本为5.19：Linux source code (v5.19) - Bootlin Elixir Cross Referencer

add_key

add_key定义在keyctl.c - security/keys/keyctl.c

接下来进行逐部分分析学习：

__user 修饰符的数据意味着来自用户空间，不能再内核空间直接解引用

  • 这里定义了函数要用到的变量，并限制负载大小为 1MB - 1B
  • 将用户态的 _type 数据拷贝到内核态的局部变量 type[32]
  • 这一部分用于处理 description 数据
  • #define KEY_MAX_DESC_SIZE 4096 ：description 最大长度为4096
  • 使用strndup_user函数从用户态 _description 拷贝数据，实际上申请了一块大小为4096的内存临时存放 description数据
  • 一系列检查与安全校验，如不能以"."开头等
  • 这一部分用于处理负载数据 payload
  • 使用 kvmalloc 申请一片内存，大小为传入的参数 plen
  • 然后从用户态 _payload 拷贝到内核态 payload
  • 一系列检查不通过后将跳转到错误处理
  • 查找可写的密钥环，才可以往里面添加密钥
  • 创建或更新密钥：
    • 匹配到已有密钥 → 更新
    • 未匹配到 → 创建新密钥
    • 自动关联到目标密钥环
  • 差错控制
  • 动态申请的内存 payload 和 description 进行释放

payload 和 description 在这一层函数的作用是临时存储用户态传递过来的数据

具体看分配内存的类型

  • GFP_USER: 分配供用户空间使用的内核内存

  • __GFP_NOWARN: 分配失败不打印内核警告

  • 来自 kmalloc 通用内核池

接下来看 kvmalloc(plen, GFP_KERNEL)，该函数实际上会依据申请内存的大小选择以 kmalloc 或者 vmalloc 的形式进行内存申请

  • 大内存分配时，先尝试 kmalloc 并且设置 GFP_FLAG ：
    • 不打印警告
    • 不无限重试（因为失败了可以回退 vmalloc）
    • 不强制必须成功（成功与否交给 vmalloc 兜底）
  • 若失败则回退到 vmalloc

kvmalloc_node 优先以 GFP_KERNEL 尝试 kmalloc 分配物理连续内存，失败则自动回退到 vmalloc 分配虚拟连续内存，是内核的 “智能万能分配器”

key_create_or_update

description长期存储内存申请

代码比较多，这里看最重要的。经过一系列检查与查找，最终申请内存创建新密钥：

该函数中会完成代表单个密钥的 key 结构体与 description 空间的分配，其中 key 结构体来自独立的 key_jar。description 的空间则使用 kmemdup() 进行分配，该函数本质上等于 slab_alloc_node() + memcpy()，可以直接理解为使用 kmalloc(size, GFP_KERNEL) 分配了一个内核对象并写入了一个字符串 description

payload长期存储内存申请

payload 通过上述函数链 kmalloc 了一块内存实现长久的存储。最终通过ret = key->type->instantiate(key, prep);存储到 key 中

总的来说经历了以下四次 kmalloc

分配作用 分配函数 GFP 标志 内存来源
临时 description kmalloc GFP_USER | __GFP_NOWARN ZONE_NORMAL
临时 payload kvmalloc GFP_KERNEL ZONE_NORMAL / vmalloc
长久 key 结构体 kmem_cache_zalloc GFP_KERNEL key_jar slab
最终 payload kmalloc GFP_KERNEL ZONE_NORMAL

总的来说，构建内核密钥时需要传入 description 和 payload 两个参数（这里只考虑 type 为 "user" 的情况）。为了在内核态安全的保存这来自用户态的数据，经历了多次内核内存申请-释放的过程：

  • 按照最大的空间申请临时内存存放 description：description = strndup_user(_description, KEY_MAX_DESC_SIZE);，方式是kmalloc(size, GFP_KERNEL) ，申请大小为 4096
  • 为 payload 申请临时空间：payload = kvmalloc(plen, GFP_KERNEL);，申请大小为传入数据
  • 进入key_create_or_update 函数：key_ref = key_create_or_update(keyring_ref, type, description, payload, plen, KEY_PERM_UNDEF, KEY_ALLOC_IN_QUOTA)
    • 为 description 申请内存长期保存数据：key_alloc 函数中 key->index_key.description = kmemdup(desc, desclen + 1, GFP_KERNEL);，申请大小为 description 长度 +1
    • 为 payload 申请内存长期保存数据：ret = index_key.type->preparse(&prep); ，upayload = kmalloc(sizeof(*upayload) + datalen, GFP_KERNEL);，申请大小为 “结构体user_key_payload的大小 + 传入数据的大小”
  • 释放临时存储 payload 的内存：kvfree_sensitive(payload, plen);
  • 释放临时存储 description 的内存：kfree(description);

keyctl

keyctl() 系统调用为我们提供了对内核中密钥的管理，其核心主要是一个巨大的 switch，类似于 ioctl，根据我们传入的不同的 option 进行不同的操作，我们主要关注以下几个：

KEYCTL_REVOKE - 释放 payload

该选项对应调用的是 keyctl_revoke_key()，其中会调用到 key_revoke()，其中会调用 key->type->revoke(key)，对于 type 为 "user" 的 key 而言最后调用到 user_revoke()：

这里会通过 call_rcu() 调用到 user_free_payload_rcu()，将 payload 释放掉：

KEYCTL_UPDATE - 更新 payload 内容

该选项会调用到 keyctl_update_key()，首先会分配一个临时对象从用户空间拷贝数据，之后调用 key_update() 更新 payload，最后释放掉临时对象：

在 key_update() 中会调用 key->type->preparse(&prep) 分配新 payload 空间并进行数据拷贝，之后调用 key->type->update(key, &prep) 更新 payload 并释放旧的 payload，最后调用 key->type->free_preparse(&prep) 来了一个“假动作”

对于 type 为 "user" 而言 preparse 指针应当为 user_preparse，其会为我们的 payload 再分配一个带有一个 user_key_payload 结构体作为头部的对象来保存我们传入的 payload，分配 flag 为 GFP_KERNEL：

对于 type 为 "user" 而言 update 指针应当为 user_update，主要就是将新的 payload 给到 key，调用 user_free_payload_rcu() 释放旧的 payload：

对于 type 为 "user" 而言 free_preparse 指针应当为 user_free_preparse，这里只是一个简单的释放操作，但传入的参数 prep->payload.data[0] 在 user_update 中已经被设为 NULL，所以这一步并没有实际作用

KEYCTL_READ - 读取 payload 内容

该选项对应调用的是 keyctl_read_key()，首先会先分配一个临时对象，之后调用 __keyctl_read_key() 将payload 拷贝到临时对象上，最后从临时对象上拷贝数据到用户空间后释放该临时对象：

__keyctl_read_key() 的主要逻辑便是调用 key->type->read(key, buffer, buflen);，对于 type 为 "user" 而言应当为 user_read()，本质上就是一个 memcpy()：
